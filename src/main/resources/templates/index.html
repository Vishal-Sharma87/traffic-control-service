A distributed, fault-tolerant job processing system with priority scheduling, heartbeat-based crash detection, automatic recovery, and a Dead Letter Queue. Built with Java 21, Spring Boot, Redis, and MySQL.

Submit a Job → Poll Result

How It Works

1

Client submits a job

POST to /paid/submit, /unpaid/submit, or /public/submit. Returns HTTP 202 + a jobId immediately — no waiting.

2

Job enters the priority queue

Stored in a Redis ZSET. Score = tier × BASE + arrivedAt. PAID jobs always pop before UNPAID, which pop before PUBLIC. Ties broken by arrival time.

3

Worker picks up and processes

A background worker pops the lowest-score job, starts a heartbeat, processes it, and saves the result to MySQL.

4

Recovery if something goes wrong

Two independent schedulers scan the processing store. If a heartbeat stops or a job takes too long — it is retried automatically. After exhausting retries, it goes to the Dead Letter Queue.

5

Client polls for result

Use the jobId from step 1 to poll GET /result/poll?jobId= until status is COMPLETED or FAILED.

Job Tiers

Tier Endpoint Max Retries Heartbeat Timeout Max Processing Time
PAID /paid/submit 5 15s 50s
UNPAID /unpaid/submit 2 6s 20s
PUBLIC /public/submit 1 3s 10s

Key Design Decisions

Atomic Lua Scripts

All critical Redis operations (enqueue, dequeue, recovery) run inside Lua scripts. Redis executes Lua atomically — no check-then-act race conditions.

Two Processing ZSETs

One ZSET tracks heartbeat expiry. Another tracks max processing time. Each scheduler queries its own ZSET — clean separation of crash vs stuck detection.

Retry Preserves Fairness

Retried jobs reuse the original arrivedAt timestamp. Score is identical to first submission — no unfair queue-jumping.

Exponential Backoff

Schedulers double their delay on Redis failures (up to 60s ceiling) and halve on success. Self-heals without any external watchdog.
